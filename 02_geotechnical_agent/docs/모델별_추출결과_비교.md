## 0. 왜 기존 OCR이 아닌 LLM Vision을 사용했는가

### 기존 OCR의 한계

시추주상도는 일반 문서와 달리 **텍스트 + 표 + 그래픽 + 색상 마킹**이 하나의 페이지에 혼재된 특수 도면입니다. 기존 OCR(Tesseract, PaddleOCR 등)은 이미지에서 **텍스트만 추출**할 수 있어, 주상도의 핵심 정보를 제대로 처리할 수 없습니다.

| 주상도 요소 | 기존 OCR | LLM Vision |
|---|---|---|
| 헤더 텍스트 (공번, 좌표 등) | ✔ 가능 | ✔ 가능 |
| 표 구조 (행/열 매핑) | ✘ 불가 (텍스트만 나열) | ✔ 표 구조 인식 |
| 지층 기둥 그래픽 (패턴/해칭) | ✘ 불가 | ✔ 이미지로 인식 |
| N치 막대 그래프 | ✘ 불가 | ✔ 그래픽 해석 |
| 빨간색 터널 마킹 | ✘ 불가 (색상 무시) | ✔ 색상 인식 |
| 심도(Depth) 기준 데이터 매핑 | ✘ 불가 (별도 코딩 필요) | ✔ 문맥으로 매핑 |
| JSON 구조화 출력 | ✘ 불가 (후처리 필요) | ✔ 프롬프트로 직접 지정 |

### LLM Vision 방식의 장점

기존 OCR은 "글자를 읽는" 도구이지만, LLM Vision은 "이미지를 이해하는" AI입니다.

```
[기존 OCR 방식]
이미지 → OCR(텍스트 추출) → 후처리 코드(표 구조 복원) → 매핑 코드 → 최종 데이터
         ↑ 4~5단계 필요, 업체별 양식마다 코드 수정 필요

[LLM Vision 방식]
이미지 → LLM Vision(텍스트+구조+의미 동시 추출) → 최종 데이터
         ↑ 1단계, 프롬프트만 변경하면 다양한 양식 대응 가능
```

- **개발 비용 절감**: OCR + 표 파싱 + 매핑 코드를 별도 개발할 필요 없이, 프롬프트 하나로 처리
- **양식 유연성**: 업체별로 주상도 양식이 달라도 프롬프트 수정만으로 대응 가능
- **복합 정보 처리**: 텍스트, 그래픽, 색상, 구조를 동시에 해석하여 심도(Depth) 기준 데이터 매핑까지 수행

### 본 비교의 목적

위와 같은 이유로 **LLM Vision API 기반 추출**을 채택하였으며, 본 보고서는 주요 4개 LLM 모델(Claude Opus, Gemini Flash, Gemini Pro, GPT-4o)의 시추주상도 데이터 추출 정확도와 비용을 비교하여 최적 모델을 선정하기 위해 작성되었습니다.

### SPT N치 — 모든 API 모델이 실패

| 현상 | 원인 분석 |

| GPT-4o: N=6→7→8→9→10→11 단조증가 날조 | N치 그래픽(막대)을 눈금으로 오독, 실제 수치 텍스트를 무시 |
| Gemini Flash/Pro: 0.5~1m 간격 보간 | 측정점 사이를 선형보간하여 가짜 데이터 생성 |
| 반발(50/18) 미인식 (GPT-4o) | "50/18" 표기를 N=50+관입18cm으로 파싱 못함 |

> N치는 지반 설계의 **가장 핵심적인 입력값**이므로 이 오류는 치명적입니다.

### TCR/RQD — 정확도 검증 한계

| 현상 | 원인 분석 |
|---|---|
| 모든 모델 간 RQD 값 불일치 | 원본 PDF의 컬럼 경계 불명확, 인접 칸 수치와 혼동 |
| Claude: 일부 미기재 | 불확실한 값은 채우지 않음 (환각 방지, 데이터 불완전) |
| 타 모델: 전부 채움 | 불확실해도 값을 생성 (환각 위험, 데이터 완전) |

> TCR/RQD는 원본 스캔 품질 개선 또는 고해상도 이미지 제공 시 정확도 향상 가능
