## 1. 검토 배경 및 목적

### 1.1 검토 배경

도로부문 설계 과업에서 지반조사 보고서의 분석 및 지반정수 산정은 상당한 시간과 전문 인력을 요구하는 반복적 업무입니다. 현재 실무에서는 엔지니어가 수십~수백 페이지의 PDF 보고서를 수작업으로 읽고, 엑셀에 수치를 옮겨 적으며, 경험식을 적용하여 설계정수를 산정하고 있습니다. 이 과정에서 단순 전사 오류, 경험식 적용 착오, 시추공 간 데이터 불일치 등의 문제가 빈번하게 발생합니다.

본 기획서는 이러한 업무를 AI Agent로 자동화하여 **데이터 추출 정확도 95% 이상**, **업무시간 70% 단축**을 달성하겠다는 목표를 제시하고 있습니다.

### 1.2 검토 목적

본 보고서는 기획된 시스템의 **기술적 구현 가능성**을 객관적으로 평가하여, 개발 의사결정에 필요한 정보를 제공하는 것을 목적으로 합니다. 구체적으로 다음 사항을 검토합니다:

- 각 모듈별 기술적 난이도와 성공 가능성
- 잠재적 리스크와 제약사항
- 목표 KPI의 현실성
- 필요 리소스와 일정의 적정성
- 실무 적용을 위한 개발 전략 권고

### 1.3 검토 관점

본 검토는 **소프트웨어 아키텍처 관점**에서 수행되었습니다. 기획서에 명시된 소스 코드를 전수 검토한 결과, 현재 **모든 기능 모듈이 껍데기(스텁) 상태**임을 확인하였습니다. 즉, 데이터 구조와 설정 파일은 잘 정의되어 있으나, 실제 동작하는 코드는 아직 한 줄도 작성되지 않은 상태입니다. 따라서 본 검토는 **기획 단계의 타당성 평가**에 해당합니다.

## 2. 기획 내용 요약

### 2.1 시스템 개요

제안된 시스템은 **지반조사 보고서(PDF)를 입력받아, 구조화된 데이터 추출 -> 지반정수 자동 산정 -> 보고서 자동 생성**까지를 일괄 처리하는 AI 기반 자동화 도구입니다.

비유하자면, 현재 엔지니어가 보고서를 읽고 엑셀에 정리하는 작업을 **디지털 조수**가 대신 수행하되, 최종 판단과 검증은 엔지니어가 하는 구조입니다.

**처리 파이프라인 흐름도:**

```
[원천 데이터: PDF 지반조사 보고서]
         |
         v
[1단계] PDF 파싱 모듈 ---- "보고서의 표와 숫자를 컴퓨터가 읽을 수 있는 형태로 변환"
         |                    (주상도, 시험성과표에서 수치 추출)
         v
[2단계] 데이터 통합 모듈 -- "시추공번호+심도 기준으로 시험결과를 짝지어 정리"
         |                    (BH-1의 심도 3m 시험결과를 해당 지층과 연결)
         v
[3단계] AI Agent 모듈 ----- "경험식 적용, 통계 처리, 이상치 탐지를 자동 수행"
         |                    (Claude AI가 작업 순서를 판단하고 도구를 호출)
         v
[4단계] 보고서 생성 모듈 -- "엑셀 요약표, 그래프, Word 보고서를 자동 생성"
                              (N치 분포도, e-log P 곡선, 지반정수 요약표 등)
```

### 2.2 주요 기능 (4개 모듈)

| 모듈 | 기능 설명 | 핵심 산출물 |
|------|----------|------------|
| **1. PDF 파싱** | 주상도/시험성과표 PDF에서 지층명, 심도, N치, 시험결과 등 수치 데이터를 자동 추출 | 구조화된 시추공 데이터 (JSON) |
| **2. 데이터 통합** | 시추공번호+심도 기준으로 주상도-시험결과를 매칭하고, USCS/AASHTO 분류 수행 | 통합 데이터셋, 지반분류 결과 |
| **3. AI Agent** | Claude AI 기반으로 분석 워크플로를 자동 실행 (경험식 적용, 통계 처리, 이상치 탐지) | 설계분야별 지반정수 산정 결과 |
| **4. 보고서 생성** | 분석 결과를 엑셀 요약표, 6종 표준 그래프, Word 보고서로 자동 출력 | 엑셀, Word, 그래프 파일 |

**대상 설계 분야:** 비탈면(토사/암반), 기초, 흙막이, 터널, 연약지반 (총 6개 분야)

### 2.3 기대 효과 (KPI)

| KPI | 목표값 | 현행 수준 (추정) |
|-----|--------|----------------|
| 데이터 추출 정확도 | 95% 이상 | 수작업 전사 시 95~98% (숙련자 기준) |
| 업무시간 단축율 | 70% | 시추공 10공 기준 약 3~5일 소요 |
| 단위 테스트 커버리지 | 80% 이상 | - |

## 3. 기술적 구현 가능성 평가

### 3.1 PDF 파싱 모듈 (구현 난이도: **상**)

이 모듈은 전체 시스템의 **가장 핵심적이고 가장 어려운 부분**입니다. PDF에서 데이터를 정확히 추출하는 것이 후속 모든 분석의 품질을 결정합니다.

#### 3.1.1 문제의 본질

지반조사 보고서 PDF는 **사람이 읽기 위해 만들어진 문서**이지, 컴퓨터가 읽기 쉬운 구조가 아닙니다. 특히 주상도(시추주상도)는 다음과 같은 특성을 가집니다:

- **그래픽 기반 레이아웃**: 표준 표(테이블) 형식이 아니라, 그림과 숫자가 혼합된 시각적 레이아웃
- **업체별 양식 차이**: 국내에 50개 이상의 지반조사 업체가 있으며, 각 업체마다 주상도 양식이 다름
- **한글/영문/숫자 혼합**: "실트질 모래", "SM", "15.3m", "N=25" 등이 하나의 페이지에 혼재
- **스캔본 존재**: 오래된 보고서는 종이 보고서를 스캔한 이미지 PDF가 상당수

#### 3.1.2 기술적 평가

| 항목 | 평가 | 비고 |
|------|------|------|
| 디지털 PDF (텍스트 내장) 파싱 | 조건부 가능 | 요약표(Summary Table) 형태는 80~85% 정확도 가능 |
| 주상도 그래픽 레이아웃 파싱 | 매우 어려움 | pdfplumber 단독으로는 30~50% 실패 예상 |
| 스캔본 PDF OCR | 제한적 가능 | 한국어 OCR 정확도 60~70%, 숫자 혼합 시 더 낮아짐 |
| 업체별 양식 대응 | 점진적 확대 필요 | 현재 설정파일(column_aliases.yaml)에 필드당 8~10개 변형만 정의 |

#### 3.1.3 핵심 리스크

**리스크 1: 주상도 레이아웃 다양성**

기획서에서 사용 예정인 pdfplumber 라이브러리는 **표 구조가 명확한 PDF**에서는 효과적이나, 주상도처럼 **그래픽 요소와 텍스트가 혼합된 복잡한 레이아웃**에서는 정확도가 크게 떨어집니다. 비유하자면, 엑셀 파일을 읽는 것과 손으로 그린 도면을 읽는 것의 차이입니다.

**리스크 2: 스캔본 대응**

pytesseract(OCR 엔진)의 한국어 인식률은 일반 텍스트에서도 80% 수준이며, 지반조사 보고서처럼 **전문 용어 + 숫자 + 단위**가 혼합된 문서에서는 60~70%까지 떨어집니다. "실트질" -> "설트질", "N=25" -> "N-25" 같은 오인식이 빈번합니다.

#### 3.1.4 권장 대안

기획서에는 포함되어 있지 않지만, **Claude Vision API를 PDF 파싱의 1차 수단으로 활용**하는 것을 강력히 권고합니다:

- PDF 페이지를 이미지로 변환한 후, Claude Vision에 "이 주상도에서 지층명, 심도, N치를 추출해주세요"라고 요청
- 기존 OCR 대비 맥락 이해 능력이 뛰어나 정확도 향상 기대
- 단, API 비용 증가 (페이지당 약 $0.01~0.05)와 처리 속도 저하 감안 필요

또한, **요약표(Summary Table) 우선 파싱 전략**을 권고합니다:
- 대부분의 지반조사 보고서에는 주상도 원본 외에 요약 정리표가 포함됨
- 요약표는 표 구조가 명확하여 pdfplumber로도 높은 정확도 달성 가능
- 주상도 직접 파싱은 요약표가 없는 경우에 대한 대안으로 위치 설정

### 3.2 데이터 통합 및 분류 모듈 (구현 난이도: **중**)

#### 3.2.1 기술적 평가

| 항목 | 평가 | 비고 |
|------|------|------|
| 시추공-시험결과 매칭 | 구현 가능 | +-0.5m 허용오차 매칭은 표준적 알고리즘 |
| USCS 분류 엔진 | 구현 가능 | 결정 트리 로직으로 구현 가능, 기준이 명확 |
| AASHTO 분류 | 구현 가능 | USCS와 유사한 결정 트리 구조 |
| RMR/SMR 산정 | 부분 가능 | 일부 입력 항목(절리 간격, 방향)은 PDF 자동 추출이 어려움 |

#### 3.2.2 주의사항

- **시추공번호 정규화 문제**: 같은 시추공을 "BH-1", "BH-01", "B-1", "시추공-1" 등 다양하게 표기하는 경우가 있으며, 이를 동일 시추공으로 인식하는 로직이 현재 기획에 누락되어 있습니다.
- **얇은 지층 매칭**: +-0.5m 허용오차가 대부분의 경우 적절하지만, 0.3m 두께의 얇은 렌즈상 지층에서는 잘못된 매칭이 발생할 수 있습니다. 지층 경계를 고려한 매칭 로직 보완이 필요합니다.
- **복수 시료 처리**: 동일 심도에서 여러 시험을 수행한 경우(예: 직접전단 + 삼축압축) 결과를 어떻게 통합할지에 대한 정책이 기획서에 명시되어 있지 않습니다.
- **RMR 산정의 한계**: RMR의 5개 평가 항목 중 "암석강도(qu)"와 "RQD"는 자동 추출이 가능하나, "절리 간격", "절리 상태", "지하수 조건"은 PDF에서 자동 추출이 어려워 엔지니어의 수동 입력이 필요할 수 있습니다.

### 3.3 AI Agent 모듈 (구현 난이도: **중**)

#### 3.3.1 아키텍처 평가

기획서에서 제안한 **Claude API Tool-use 방식**의 Agent 아키텍처는 본 과업의 규모에 적합합니다. AI Agent가 "어떤 도구를 어떤 순서로 사용할지" 판단하는 오케스트레이션 역할을 수행하고, 실제 계산은 사전 정의된 도구 함수가 처리하는 구조입니다.

비유하자면, AI Agent는 **작업 지시를 내리는 팀장** 역할이고, 실제 계산은 **각 팀원(도구 함수)**이 수행하는 것입니다. 팀장이 모든 계산을 직접 하지 않아도 되므로, AI의 약점(수치 계산 부정확성)을 보완할 수 있습니다.

#### 3.3.2 핵심 리스크: AI 환각(Hallucination)

| 우려 사항 | 심각도 | 대응 방안 |
|----------|--------|----------|
| AI가 잘못된 경험식을 적용 | 높음 | 경험식 선택을 AI가 하지 않고, 토질 유형별 매핑 테이블로 결정 |
| AI가 수치 계산을 직접 수행 | 높음 | 모든 수치 계산은 반드시 Python 코드로 구현 (AI는 호출만) |
| AI가 존재하지 않는 데이터를 생성 | 중간 | 도구 함수의 반환값만 사용하도록 프롬프트에 엄격히 제한 |
| 설계분야별 적용 기준 혼동 | 중간 | 설계분야를 명시적 입력으로 받아 분기 처리 |

**핵심 원칙**: AI Agent는 **작업 흐름을 조율하는 역할**만 수행하고, **공학적 계산은 100% 결정론적 코드**로 구현해야 합니다. 이 원칙이 지켜지지 않으면 엔지니어가 AI 산출물을 신뢰할 수 없게 됩니다.

#### 3.3.3 경험식 설정의 문제점

현재 기획서의 경험식 설정(`empirical_formulas.yaml`)에서 다음 문제가 확인되었습니다:

- **비배수전단강도 경험식 단위 오류 가능성**: Terzaghi-Peck의 `Su = N/8`은 원래 **tsf(톤/ft2)** 단위로 제안된 것인데, 설정 파일에는 **(kPa)** 로 표기되어 있습니다. kPa로 해석하면 N=20일 때 Su=2.5 kPa가 되어 공학적으로 비현실적인 극소값이 됩니다. 현행 실무에서는 **Hara et al.의 `Su = 29 * N^0.72` (kPa)**가 더 적합합니다. 복수 경험식을 제공하고 엔지니어가 선택할 수 있어야 합니다.
- **누락된 경험식**: 말뚝 지지력 산정(Meyerhof 말뚝, 한국 도로교 설계기준), 수평지반반력계수(kh), 암반 변형계수(Hoek-Brown), 압밀계수(Cv) 추정식 등이 빠져 있습니다.
- **설정 파일과 프롬프트 간 불일치**: 탄성계수 경험식이 대표적 사례입니다. `empirical_formulas.yaml`에서는 **Bowles 공식 `E = 500 * (N+15) kPa`** (즉, E = 0.5*(N+15) MPa)를 정의하고 있으나, `prompts/parameter_estimation.py`에서는 **`E = 0.5~1.0 * N (MPa)`** 로 기술하고 있어 근본적으로 다른 공식입니다. N=20일 때 Bowles는 17.5 MPa, 프롬프트 공식은 10~20 MPa로 결과가 달라집니다. 어느 것이 실제 적용될지 혼란이 발생할 수 있습니다.
- **설정 파일 누락**: 기획서(CLAUDE.md)에서 참조하는 `config/design_standards.yaml` 파일이 실제 코드베이스에 존재하지 않습니다. 적용 기준(KDS, AASHTO) 정보를 관리할 설정 파일이 계획에는 있으나 아직 만들어지지 않은 상태입니다.

| 6종 표준 그래프 생성 | 구현 가능 | matplotlib로 충분히 구현 가능 |
| Excel 요약표 출력 | 구현 가능 | openpyxl 활용, 표준적 기능 |
| Word 보고서 자동 생성 | 구현 가능 | python-docx로 구현 가능하나 양식 완성도는 추가 작업 필요 |
| HWP(한글) 출력 | 구현 불가 | 한글 파일 생성을 지원하는 Python 라이브러리 없음 |
| PDF 출력 | 조건부 가능 | LibreOffice 설치 필요, 서버 환경에서 추가 설정 필요 |

#### 3.4.2 HWP 문제: 관공서 납품의 걸림돌

**관공서 납품 시 HWP(한글) 양식이 필수인 경우가 대부분**입니다. 현재 Python 생태계에서 HWP 파일을 직접 생성할 수 있는 안정적인 라이브러리가 없습니다. 이는 관공서 대상 과업에서 **배포의 실질적 장벽**이 됩니다.

대안은 다음과 같습니다:
1. Word(.docx) 출력 후, 한글 프로그램에서 수동 변환 (실무적으로 가장 현실적)
2. hwp-converter 등 비공식 도구 활용 (안정성 보장 불가)
3. 관공서에 Word 양식 제출 협의 (과업별로 가능 여부 상이)

민간 부문 과업에서는 Word/Excel 출력으로 충분하므로 즉시 활용 가능합니다.

#### 3.4.3 한글 폰트 문제

matplotlib에서 한글을 표시하려면 별도의 폰트 설정이 필요합니다. 서버 환경(Linux)에서는 한글 폰트가 기본 설치되어 있지 않아, 나눔고딕 등의 폰트를 별도로 설치하고 matplotlib 설정 파일을 수정해야 합니다. 사소하지만 빠뜨리면 그래프에서 한글이 깨져 보입니다.

## 6. 품질 목표 달성 가능성

### 6.1 데이터 추출 정확도 95% 평가

**결론: 1차 버전에서 95% 달성은 비현실적입니다.**

| PDF 유형 | 추정 정확도 (v1) | 95% 달성 가능 시점 |
|----------|-----------------|-----------------|
| 디지털 PDF (요약표 형태) | 80~85% | v2~v3 (6개월 이후) |
| 디지털 PDF (주상도 직접 파싱) | 60~70% | 상당 기간 소요 (1년 이상) |
| 스캔본 PDF | 50~65% | 매우 어려움 |

**정확도 95%란 무엇을 의미하는가:**

시추공 1개에 약 50~100개의 개별 데이터 포인트(지층명, 심도, N치, 시험값 등)가 있습니다. 95% 정확도란 이 중 97~100개가 정확해야 한다는 의미입니다. 실제로 지층명 1개가 잘못 인식되거나, N치 1개가 누락되면 정확도가 떨어집니다.

**실무적 의미:**

- 80~85% 정확도에서도 **수작업 대비 상당한 시간 절약**이 가능합니다 (엔지니어가 확인/수정하는 시간 포함)
- 핵심은 **오류를 자동으로 탐지하고 엔지니어에게 알려주는 것**입니다 (이상치 탐지 기능)
- 정확도 95%는 **장기 목표**로 설정하고, 1차 목표는 85%로 조정하는 것이 적절합니다

### 6.2 업무시간 70% 단축 평가

**결론: 전체 워크플로의 70% 단축은 가능하나, 조건이 있습니다.**

| 업무 단계 | 현행 소요시간 (시추 10공) | 자동화 후 | 단축율 |
|----------|----------------------|----------|--------|
| PDF 읽기 + 데이터 입력 | 8~12시간 | 0.5~1시간 (파싱 + 검증) | 85~95% |
| USCS/지층 분류 | 2~4시간 | 0.1시간 (자동) | 95% 이상 |
| 경험식 적용 + 정수 산정 | 4~8시간 | 0.2시간 (자동) | 95% 이상 |
| 이상치 확인 | 1~2시간 | 0.5시간 (자동 탐지 + 검증) | 50~75% |
| 보고서/그래프 작성 | 4~8시간 | 0.5~1시간 (자동 + 수정) | 80~90% |
| **합계** | **19~34시간** | **1.8~2.8시간** | **85~91%** |

**단, 이 단축율은 다음 조건이 충족될 때 적용됩니다:**
1. PDF가 디지털 형식이고 요약표가 포함된 경우
2. 해당 업체 양식이 시스템에 등록된 경우
3. 엔지니어가 자동 산출물을 검증하는 시간을 포함한 경우

> **주의**: PDF 파싱 품질이 낮으면 수정 작업에 더 많은 시간이 소요되어 오히려 비효율이 발생할 수 있습니다. 따라서 파싱 정확도가 80% 이하인 PDF에 대해서는 자동화를 적용하지 않고 수동 입력 경로를 제공해야 합니다.

## 7. 단계별 추천 개발 전략

### 7.1 MVP(최소 기능 제품) 전략

전체 시스템을 한 번에 완성하기보다, **엔지니어가 즉시 활용할 수 있는 최소 기능**부터 배포하고 점진적으로 확대하는 전략을 권고합니다.

**MVP 범위 (우선 구현):**

1. **디지털 PDF 요약표 파싱** (주상도 직접 파싱은 후순위)
   - 가장 표준화된 요약표 양식 3종부터 지원
   - 파싱 실패 시 수동 입력 인터페이스 제공

2. **USCS 분류 + 기본 경험식 적용**
   - N치 기반 내부마찰각, 단위중량 산정 (가장 빈용되는 경험식)
   - 비탈면(토사) 분야를 최우선 대상으로

3. **Excel 요약표 + N치 분포 그래프 출력**
   - 실무에서 가장 많이 사용하는 2가지 산출물부터

4. **이상치 탐지 경고**

## 8. 종합 의견 및 권고

### 8.1 종합 판정

**"조건부 개발 가능"**

본 시스템은 기술적으로 구현이 가능하며, 실무에서 상당한 업무 효율 향상을 가져올 수 있는 프로젝트입니다. 그러나 기획서의 목표치(정확도 95%, 기간 10주)는 지나치게 낙관적이며, 현실적인 기대치 조정과 단계적 접근이 필요합니다.

기획서의 강점은 **데이터 모델이 잘 정의**되어 있고, **설계분야별 파이프라인이 체계적으로 기획**되어 있다는 점입니다. 설정 파일(YAML) 기반의 확장 가능한 구조도 좋은 설계 방향입니다. 다만, **PDF 파싱의 난이도를 과소평가**하고 있으며, **AI Agent의 역할 범위를 명확히 제한**하는 설계 원칙이 기획서에 부족합니다.

### 8.2 핵심 성공 조건

아래 3가지 조건이 충족되지 않으면 프로젝트 성공이 어렵습니다:

**조건 1: 실제 데이터 기반 개발**

개발 착수 전 **최소 3개 업체, 10건 이상의 실제 지반조사 PDF**를 확보하고, 각각에 대해 엔지니어가 수동으로 추출한 정답 데이터(Ground Truth)를 작성해야 합니다. 이 데이터가 없으면 파서의 정확도를 측정할 수 없고, 어느 방향으로 개선해야 할지 판단할 수 없습니다.

**조건 2: AI 계산 의존 배제 원칙**

모든 공학적 수치 계산(경험식 적용, 통계 처리, 분류 판정)은 **결정론적 Python 코드**로 구현하고, AI Agent는 **작업 흐름 조율과 결과 해석 보조**에만 사용해야 합니다. "Claude에게 N치에서 내부마찰각을 계산해달라"는 접근은 결과의 재현성과 신뢰성을 보장할 수 없습니다. 핵심 계산 로직이 AI 없이도 동작할 수 있어야 합니다.

**조건 3: 도메인 전문가의 지속적 참여**

지반공학 전문가(설계 실무 경력 5년 이상)가 개발 초기부터 다음 역할을 수행해야 합니다:
- 경험식 선정 및 적용 조건 검증
- 이상치 탐지 기준값 설정
- 자동 산출물의 공학적 타당성 검증
- 설계기준 변경 시 업데이트 자문

### 8.3 실무자를 위한 체크리스트

본 프로젝트의 개발 착수를 검토하는 실무자를 위한 체크리스트입니다:

**착수 전 확인 사항:**

| 확인 항목 | 확인 여부 | 비고 |
|----------|----------|------|
| 실제 지반조사 PDF 10건 이상 확보 가능한가? | | 보안 정책 확인 필요 |

## 부록

### A. 기술 용어 해설 (비개발자용)

| 용어 | 설명 |
|------|------|
| **PDF 파싱** | PDF 문서에서 텍스트, 숫자, 표 등의 데이터를 컴퓨터가 처리할 수 있는 형태로 변환하는 작업. 사람이 보고서를 읽고 엑셀에 옮기는 것을 기계가 자동으로 수행하는 것과 같습니다. |
| **OCR (광학 문자 인식)** | 이미지 속의 글자를 텍스트로 변환하는 기술. 스캔한 문서를 컴퓨터가 읽을 수 있게 만듭니다. 스마트폰 카메라로 명함을 찍으면 연락처가 자동 입력되는 것과 같은 원리입니다. |
| **AI Agent** | 사용자의 지시를 받아 여러 도구를 자동으로 사용하여 작업을 수행하는 AI 프로그램. 마치 숙련된 조수가 "이 보고서 분석해줘"라는 지시를 받으면 필요한 작업을 순서대로 수행하는 것과 같습니다. |
| **Tool-use** | AI Agent가 프로그램 내의 기능(도구)을 직접 호출하여 사용하는 방식. AI가 계산기, 검색 엔진, 그래프 생성기 등을 필요에 따라 자동으로 선택하여 사용합니다. |
| **Claude API** | Anthropic 사에서 제공하는 AI 서비스를 프로그램에서 호출하는 인터페이스. 인터넷을 통해 AI의 판단 능력을 빌려 쓰는 것입니다. 사용량에 따라 비용이 발생합니다. |
| **AI 환각 (Hallucination)** | AI가 실제로는 없는 정보를 마치 있는 것처럼 생성하는 현상. 예를 들어, AI가 실제 데이터에 없는 시험 결과를 만들어내거나, 존재하지 않는 경험식을 인용하는 경우입니다. |
| **스텁 (Stub)** | 아직 내용이 채워지지 않은 빈 코드 틀. 건축에서 골조만 세워놓고 내부 마감이 안 된 상태와 유사합니다. |
| **MVP** | 최소 기능 제품(Minimum Viable Product). 핵심 기능만 먼저 만들어 실무에서 테스트하고, 피드백을 받아 점진적으로 완성하는 개발 전략입니다. |
| **pdfplumber** | PDF 파일에서 텍스트와 표를 추출하는 Python 프로그램. 표 구조가 명확한 PDF에서는 효과적이나, 그래픽이 복잡한 문서에서는 한계가 있습니다. |
| **Claude Vision** | AI가 이미지를 "보고" 내용을 이해하는 기능. 사람이 주상도를 눈으로 보고 수치를 읽는 것처럼, AI가 이미지 속 표와 숫자를 인식합니다. |
| **데이터 주권** | 데이터가 어느 나라의 서버에 저장되고 처리되는지에 관한 법적/정책적 문제. Claude API는 미국 서버에서 처리되므로, 국내 관공서 데이터 전송 시 보안 정책을 확인해야 합니다. |
| **결정론적 코드** | 같은 입력을 넣으면 항상 같은 결과가 나오는 프로그램. AI와 달리 매번 동일한 결과를 보장하므로 공학적 계산에 적합합니다. 계산기에 2+3을 입력하면 항상 5가 나오는 것과 같습니다. |
| **YAML** | 사람이 읽기 쉬운 설정 파일 형식. 경험식의 계수나 이상치 기준값 등을 프로그램 코드와 분리하여 관리함으로써, 프로그래머가 아닌 사람도 값을 변경할 수 있습니다. |
| **JSON** | 컴퓨터 프로그램 간 데이터를 교환하기 위한 표준 형식. 시추공 데이터를 구조화하여 저장하는 데 사용됩니다. |
| **경험식 (Empirical Formula)** | 이론적 유도가 아닌, 다수의 실험/현장 데이터를 통계적으로 분석하여 도출한 공식. 예: Meyerhof(1956)의 N치-내부마찰각 관계식. |
| **Ground Truth** | 정답 데이터. AI의 결과를 검증하기 위해 사람이 수동으로 확인한 정확한 값입니다. |

| 사례 | 범위 | 결과 | 시사점 |
