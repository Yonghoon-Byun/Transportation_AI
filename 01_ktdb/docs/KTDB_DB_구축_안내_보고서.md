## 1. 왜 데이터베이스가 필요한가

### 1.1 현재 작업 방식의 문제

현재 KTDB 데이터를 활용한 교통 분석 업무는 다음과 같은 과정을 거칩니다.

```
[KTDB 홈페이지] → ZIP 다운로드 → 압축 해제 → Excel/TXT 파일 열기 → 수작업 분석 → 보고서 작성
```

이 방식에는 몇 가지 실무적 어려움이 있습니다.

| 문제 | 구체적 상황 | 영향 |
|------|------------|------|
| **파일이 너무 많다** | ZIP 25건, 내부 파일 수백 개 `[A-§2]` | 원하는 데이터를 찾는 데만 상당한 시간 소요 |
| **파일 포맷이 제각각** | Excel(.xlsx), 텍스트(.OUT, .TXT) 혼재 `[A-§6]` | 권역마다 열기·파싱 방법이 다름 |
| **데이터 양이 방대** | 전체 약 **9,300만 행** `[B-§5]` | Excel로 열 수 없는 규모 (Excel 행 한계: 약 100만 행) |
| **수단 명칭이 권역마다 다름** | 같은 "지하철"이 `지하철`, `도시철도`, `미명명` `[A-§7]` | 비교·합산 시 혼란 |
| **합계 행 이중 계산 위험** | 원본에 "합계" 행이 포함되어 있어 SUM 시 2배 오류 `[C-§문제1]` | 보고서 수치 오류 가능성 |

### 1.2 데이터베이스가 해결하는 것

```
[KTDB ZIP] → 자동 파싱·정제 → [PostgreSQL 데이터베이스] → AI 자연어 질의 → 즉시 결과
```

| 개선 사항 | 설명 |
|-----------|------|
| **단일 저장소** | 25개 ZIP의 모든 데이터가 하나의 DB에 통합 |
| **자동 변환** | Excel 행렬 → DB 세로형 자동 변환 (아래 §3.2 참조) |
| **즉시 검색** | "서울→부산 출근 통행량"을 자연어로 질의하면 5초 내 응답 `[C-§5.4]` |
| **이중 계산 방지** | "합계" 행을 자동으로 구분하여 집계 오류 원천 차단 `[C-§1.1]` |
| **수단명 통일** | 동의어 매핑 테이블로 "지하철=도시철도" 자동 변환 `[C-§1.3]` |

## 2. KTDB 원본 데이터는 어떤 것인가

### 2.1 데이터 전체 분류

> 출처: `[A-§2.1]` ZIP 파일 목록 및 분류 체계

KTDB에서 배포하는 데이터는 **분류 코드 체계**로 관리됩니다.

```
코드 구조:  {연도}-OD-{대분류}-{소분류}-{권역코드}
예시:       2024-OD-PSN-OBJ-00    → 2024년, 여객, 목적별, 전국
            2024-OD-FRE-CAR-00    → 2024년, 화물, 자동차, 전국
```

| 대분류 | 소분류 | 설명 | ZIP 수 |
|--------|--------|------|--------|
| PSN(여객) | OBJ(목적별) | 출근·등교·업무·귀가·기타별 통행량 | 7건 |
| PSN(여객) | MOD-1x(수단별) | 승용차·버스·철도 등 수단별 통행량 | 7건 |
| PSN(여객) | MOD-2x(접근수단 등) | 철도역 접근수단 OD, 목적×수단 OD | 7건 |
| FRE(화물) | CAR(자동차) | 소·중·대형 화물차 톤급별 통행량 | 1건 |

### 2.2 권역별 데이터 제공 방식의 차이 (핵심)

> 출처: `[A-§3]` ZIP 내부 폴더 구조, `[A-§6]` 포맷 차이 비교표

KTDB 데이터는 **같은 종류의 데이터인데도 권역에 따라 파일 형태가 완전히 다릅니다.** 이것이 DB 구축 시 가장 큰 난이도를 높이는 요인입니다.

| 구분 | 전국 250존 | 비수도권 5개 권역 | 수도권 |
|------|-----------|-------------------|--------|
| **파일 포맷** | Excel (.xlsx) | Excel (.xlsx) | **텍스트 (.OUT, .TXT)** |
| **인코딩** | UTF-8 (Excel 기본) | UTF-8 (Excel 기본) | **CP949 (한글 전용 인코딩)** |
| **헤더(열 이름)** | 있음 | 있음 | **없음** (1행부터 데이터) |
| **존코드 형태** | 정수 1~250 | 정수 1~N | **7자리 문자열** (예: 1101072) |
| **연도 구분** | 시트별 분리 | 시트별 분리 | **파일별 분리** |
| **합계 컬럼** | 숫자 또는 수식 | **Excel 수식** | 숫자 |
| **파생 컬럼** | 없음 | **있음** (승용차/택시 합산 등) | 없음 |

> **설계자 관점에서의 의미**: 단순히 "Excel 파일을 DB에 넣으면 되겠지"라고 생각하기 쉽지만, 실제로는 권역마다 전혀 다른 파싱(읽기) 로직이 필요합니다. 특히 수도권 텍스트 파일은 열 이름이 없어서 별도 설명자료(.hwp)를 확인해야만 어떤 열이 어떤 수단인지 알 수 있습니다.

### 2.3 데이터 규모

> 출처: `[B-§5]` 전체 테이블 목록 및 볼륨

| 데이터 종류 | 예상 행 수 | 비고 |
|-------------|-----------|------|
| 전국 250존 여객 OD | 약 87만 행 | 목적별 + 수단별 |
| 비수도권 권역별 OD | 약 1,850만 행 | 5개 권역 합산 |
| 수도권 OD | 약 **7,200만 행** | 전체의 77% |
| 화물 OD | 약 44만 행 | |
| 사회경제지표 | 약 16만 행 | |
| 기타 (접근수단, 재차인원 등) | 약 140만 행 | |
| **합계** | **약 9,300만 행** | |

> **수도권 목적×수단 OD** 단일 테이블이 **4,800만 행**으로 전체 데이터의 절반 이상을 차지합니다. `[B-§11.3]`

## 3. 데이터베이스 구축 시 알아야 할 핵심 개념

### 3.1 존(Zone) 체계 — "지역을 번호로 관리한다"

> 출처: `[B-§2]` 핵심 개념 이해, `[A-§5]` 존체계 분석

KTDB는 행정구역을 **번호 체계**로 관리합니다.

```
[17존] 시도 수준        예: 1=서울, 2=부산, ... 17=제주
  └── [250존] 시군구 수준   예: 1=종로구, 2=중구, ...
       └── [소존] 읍면동 수준   수도권 1,310개, 비수도권 권역별 수백 개
```

| 존 체계 | 존 수 | 사용처 | 형태 |
|---------|-------|--------|------|
| 17존 (시도) | 17개 | 시도간 집계 분석 | 정수 1~17 |
| 250존 (시군구) | 250개 | 전국 OD, 화물 OD, 사회경제지표 | 정수 1~250 |
| 수도권 소존 (읍면동) | 1,310개 | 수도권 상세 분석 | **7자리 행정기관코드** |
| 비수도권 소존 (읍면동) | 권역별 수백 개 | 권역 내부 상세 분석 | 정수 1~N (권역마다 1번부터) |

> **주의**: 비수도권 소존 번호는 권역마다 1번부터 시작합니다. 예를 들어 부산울산권 1번과 대구광역권 1번은 전혀 다른 지역입니다. 따라서 **권역코드 + 소존번호**를 함께 사용해야 유일하게 식별할 수 있습니다. `[B-§7.3]`

### 3.2 OD 데이터 — "출발지→도착지 통행량"

> 출처: `[B-§2]` OD 데이터란

OD(Origin-Destination)는 출발지와 도착지 사이의 통행량입니다. KTDB 원본은 **행렬(매트릭스)** 형태로 제공하지만, DB에서는 **세로형(Tall)** 으로 변환하여 저장합니다.

```
[원본 엑셀: 행렬 형태]                    [DB 저장: 세로형]
        도착1  도착2  도착3 ...            출발존 | 도착존 | 연도 | 승용차 | 버스 ...
출발1    100    200    150 ...               1   |   1   | 2023 |  100  | 200
출발2    300     50    400 ...               1   |   2   | 2023 |  300  |  50
출발3    ...                                 1   |   3   | 2023 |  ...
```

이렇게 변환하면 **"2023년 종로구에서 해운대구로 가는 승용차 통행량은?"** 같은 질의를 간단한 검색으로 처리할 수 있습니다.

> **용어 참고**: 이 "세로형(Tall)" 변환을 개발 용어로는 "롱폼(Long form)"이라 하고, 원본 행렬 형태를 "와이드폼(Wide form)"이라 합니다.

### 3.3 교통수단 분류 체계의 차이 (가장 복잡한 이슈)

> 출처: `[A-§7]` 수단 분류 체계 비교, `[C-§1.3]` ktdb_mode_mapping

**같은 교통수단을 권역마다 다른 이름으로 부르는 것**이 DB 설계에서 가장 까다로운 문제입니다.

| 교통수단 | 전국 250존 표기 | 비수도권 표기 | 수도권 표기 |
|----------|----------------|--------------|------------|
| 도시철도 | `지하철` | `도시철도` | 열 이름 없음 (추정) |
| 일반+고속철도 | `일반철도`, `고속철도` (분리) | `일반/고속철도` (합산) | 열 이름 없음 (추정) |
| 도보+자전거 | **없음** | `도보/자전거` | 열 이름 없음 (추정) |
| 택시 | **없음** | `택시` | 열 이름 없음 (추정) |
| 항공 | `항공` | **없음** (기타에 포함 추정) | 불명 |
| 해운 | `해운` | **없음** (기타에 포함 추정) | 불명 |

> **해결 방법**: DB에 **"동의어 매핑 테이블"** (`ktdb_mode_mapping`)을 만들어서, "지하철"과 "도시철도"가 같은 수단임을 기록합니다. AI가 "지하철 통행량"을 질의하면 전국 OD에서는 `mode='지하철'`, 권역 OD에서는 `mode='도시철도'`로 자동 변환합니다. `[C-§1.3]`

### 3.4 파생 행(합계 행)의 이중 계산 문제

> 출처: `[C-§문제1, §1.1]`

원본 데이터에는 "합계" 행이 원본 행과 함께 저장되어 있습니다. 만약 DB에서 `SUM(통행량)`을 계산하면 **원본 값 + 합계 값이 더해져서 2배가 되는 오류**가 발생합니다.

```
목적     | 통행량
---------|--------
출근     |  100    ← 원본
등교     |   50    ← 원본
합계     |  150    ← 파생 (출근+등교의 합)
---------|--------
SUM 결과 |  300    ← ❌ 실제는 150이어야 함
```

> **해결 방법**: 모든 OD 행에 `is_derived` (파생 여부) 컬럼을 추가합니다. "합계" 행은 `is_derived=TRUE`로 표시하고, AI가 사용하는 분석 뷰(View)에서는 `is_derived=FALSE`인 원본 행만 자동으로 가져옵니다. `[C-§1.1, §2.1]`

## 4. 구축된 DB의 전체 구조

### 4.1 데이터 처리 흐름

> 출처: `[B-§3]` 원본 데이터 → DB 테이블 매핑

```
[KTDB ZIP 25건 다운로드]
         │
         ▼
[자동 압축 해제 및 파싱]
    ┌────┼────┬──────────┐
    ▼    ▼    ▼          ▼
  .OUT  .TXT  .xlsx     .xlsx
 (수도권) (수도권) (비수도권)  (전국250존)
  CP949   CP949  UTF-8     UTF-8
    │     │      │         │
    └─────┴──────┴─────────┘
              │
    [데이터 정제 및 형식 통일]
    · 행렬 → 세로형 변환
    · 인코딩 통일 (CP949 → UTF-8)
    · 수단명 표준화
    · 파생 행(합계) 표시
              │
              ▼
      [PostgreSQL 데이터베이스]
       약 9,300만 행 적재
```

### 4.2 DB 테이블 구성 요약

> 출처: `[B-§5]` 전체 테이블 목록, `[C-§2.6]` LLM 노출 대상

DB는 크게 5가지 종류의 테이블로 구성됩니다.

| 종류 | 대표 테이블 | 역할 | 행 수 |
|------|-------------|------|-------|
| **참조 테이블** (6개) | `ref_region`, `ref_year`, `ref_purpose` 등 | 코드(번호)의 이름과 속성 관리 | 수십 행 |
| **존체계 테이블** (3개) | `zone_250`, `zone_emd_metro`, `zone_emd_region` | 존 번호 ↔ 행정구역 이름 매핑 | ~3,800행 |
| **OD 테이블** (10개) | `od_national_purpose`, `od_metro_mode` 등 | **핵심 통행량 데이터** | ~9,200만 행 |
| **사회경제지표** (6개) | `socio_national`, `socio_metro_pop` 등 | 인구·종사자수·학생수 등 | ~16만 행 |
| **운영 테이블** (2개) | `etl_file_meta`, `etl_log` | 데이터 적재 이력 관리 | 수십~수백 행 |

### 4.3 테이블 간 관계 (간략 ERD)

> 출처: `[B-§16]` 테이블 간 관계

```
[참조 테이블: 권역코드, 연도, 목적코드, 수단코드]
         │
         ▼ (코드 참조)
[존체계 테이블]  ←──── "종로구는 250존 1번" 같은 매핑 정보
    │
    ▼ (존번호로 연결)
[OD 테이블]  ←───── "1번 존 → 34번 존, 출근, 100통행/일"
    │
    ▼ (같은 존번호로 연결)
[사회경제지표]  ←── "1번 존, 2023년, 총인구 = 123,456명"
```

**실무적 의미**: 존체계 테이블이 "연결고리" 역할을 합니다. OD 데이터에는 존 번호만 있고, 존체계 테이블에서 "그 번호가 어디인지"를 찾습니다. 이 구조 덕분에 "종로구→해운대구 출근 통행량"처럼 지역 이름으로 검색이 가능합니다.

### 4.4 대용량 데이터 처리 전략: 파티셔닝

> 출처: `[B-§17]` 인덱스·파티셔닝 전략, `[C-§1.2]` 파티션 전략 수정

**파티셔닝**이란 대용량 테이블을 물리적으로 여러 조각으로 나누어 저장하는 방식입니다.

```
예시: 수도권 OD 1,200만 행을 연도별로 분할

[od_metro_mode 전체]
    ├── 2023년 파티션 (~170만 행)
    ├── 2025년 파티션 (~170만 행)
    ├── 2030년 파티션 (~170만 행)
    ├── 2035년 파티션 (~170만 행)
    ├── 2040년 파티션 (~170만 행)
    ├── 2045년 파티션 (~170만 행)
    └── 2050년 파티션 (~170만 행)
```

**효과**: "2023년 수도권 통행량"을 검색하면 2023년 파티션(170만 행)만 읽으면 되므로, 전체 1,200만 행을 읽는 것보다 **7배 빠릅니다**.

| 테이블 | 파티션 기준 | 파티션 수 | 이유 |
|--------|-----------|----------|------|
| 수도권 OD 3종 | 연도(year) | 각 7개 | 1,200만~4,800만 행, 연도별 조회 빈번 |
| 비수도권 OD 2종 | 권역(region_code) | 각 5개 | 권역별 분석 쿼리 빈번 |
| 전국 250존 OD 3종 | 연도(base_year), **RANGE 방식** | 각 7개 | 연도 범위 조회 최적화 `[C-§1.2]` |

> **RANGE vs LIST 파티션**: 전국 OD는 "2023~2030년 추이" 같은 범위 조회가 많아서, RANGE 방식(연속 범위 스캔)이 LIST 방식(개별 목록 스캔)보다 효율적입니다. `[C-§1.2]`

## 5. AI(자연어 질의) 연동 방식

### 5.1 작동 원리: Text-to-SQL

> 출처: `[C-§3.1, §3.5]` Text-to-SQL 방식 선택 근거, LLM 응답 파이프라인

AI 연동의 핵심은 **자연어를 SQL(데이터베이스 질의어)로 자동 변환**하는 것입니다.

```
사용자 입력: "2023년 서울에서 부산으로 출근 통행량은?"
    │
    ▼
[AI가 자동으로 SQL 생성]
    SELECT origin_sido, dest_sido, trips
    FROM mv_od_purpose_17
    WHERE origin_sido = '서울특별시'
      AND dest_sido = '부산광역시'
      AND base_year = 2023
      AND purpose = '출근'
    │
    ▼
[DB에서 실행하여 결과 반환]
    서울특별시 → 부산광역시, 2023년, 출근, 12,345 통행/일
    │
    ▼
[AI가 결과를 해석하여 표+텍스트로 제공]
    "2023년 기준 서울에서 부산으로의 출근 통행량은
     일평균 12,345통행(AAWDT 기준)입니다."
```

### 5.2 AI 전용 뷰(View) 레이어

> 출처: `[C-§2]` LLM 전용 뷰 레이어 설계

AI가 직접 원본 테이블을 보지 않고, **"분석 뷰"** 라는 가공된 창을 통해서만 데이터를 봅니다.

| 뷰(View) | 내장 기능 | 사용 예시 |
|-----------|----------|----------|
| `v_od_purpose_250` | 합계 행 자동 제외 + 지역 이름 자동 연결 | "종로구→해운대구 출근 통행량" |
| `v_od_mode_250` | 합계 행 자동 제외 + 수단 동의어 매핑 | "2023년 전국 승용차 통행량" |
| `mv_od_purpose_17` | **시도 수준 사전 집계** (즉시 응답) | "서울→부산 출근 통행량" |
| `v_zone_lookup` | 지역 이름 → 존 번호 검색 | "해운대구가 몇 번 존인지" |

**뷰의 장점**:
- 이중 계산 방지가 **자동으로** 적용 (is_derived=FALSE 필터 내장)
- 지역 이름이 **자동으로** 붙어서 나옴 (존 번호만으로는 어디인지 모름)
- AI가 잘못된 테이블에 접근하는 것을 **원천 차단**

### 5.3 성능 목표

> 출처: `[C-§5.4]` 성능 KPI

| 단계 | 목표 시간 | 달성 전략 |
|------|----------|----------|
| AI가 SQL 생성 | 2~3초 | LLM 모델 활용 |
| DB에서 쿼리 실행 | 1~2초 이내 | 파티셔닝 + 인덱스 + Materialized View |

| **전체 (end-to-end)** | **5초 이내** | |

## 7. 한계점과 주의사항

### 7.1 데이터 자체의 한계

> 출처: `[A-§4, §7, §8]`, `[C-§문제점 요약]`

| 한계 | 상세 내용 | 영향 | 대응 방안 |
|------|----------|------|----------|
| **수도권 TXT 열 이름 부분 미확인** | col5~col14 중 7개는 비수도권 대응·값 크기로 추정 완료(도보/자전거, 승용차, 버스, 일반철도, 고속철도, 도시철도, 택시), **col12~14 3개는 여전히 불명** `[A-§4.4, §7.1]` | 추정값 오류 시 분석 결과에 영향 | 설명자료(.hwp)로 추정값 검증 및 불명 3개 확인 필요 `[A-§8.1 C-1]` |
| **수도권 "업무" 목적 누락** | 목적별 주수단 OD(MOD-21)에 업무 파일이 없음 `[A-§4.7]` | 수도권 업무 목적의 수단별 분석 불가 | KTDB에 의도적 누락 여부 확인 필요 `[A-§8.1 C-2]` |
| **수단 분류 체계 3종 혼재** | 전국/비수도권/수도권 각각 다른 수단 분류 `[A-§7]` | 전국 단위 통합 분석 시 완벽한 비교 불가 | 동의어 매핑 테이블로 부분 해결 `[C-§1.3]` |
| **사회경제지표 권역별 차이** | 인구 시트명이 `5-24세` vs `5-29세`로 권역마다 다름 `[B-§8.2]` | ETL에서 일관된 지표 코드 매핑 필요 | 지표코드 매핑 규칙 정의 완료 `[B-§6.6]` |
| **화물 OD 컬럼명 오류** | 도착 17존 컬럼명이 출발과 동일 (`대존O_17` 중복) `[A-§4.8]` | 위치 기반 파싱 필수 | ETL 파서에서 위치 기반 처리 `[A-§8.2 D-6]` |

> **보고서 작성 시 단위 주의**: DB에는 단위 환산을 위한 부가 테이블이 포함됩니다. `[B-§14]`
> - **재차인원 테이블** (`occupancy_rate`): 여객 OD 통행량(통행/일)을 차량 대수(대/일)로 환산할 때 사용합니다. 예: 승용차 재차인원 1.3명 → 100통행은 약 77대.
> - **AADT 전환계수 테이블** (`aadt_factor`): 주중 평균(AAWDT)을 연평균(AADT)으로 환산할 때 사용합니다. 보고서에 "연평균 일통행량"을 기재해야 하는 경우 이 계수가 필요합니다.

### 7.2 기술적 한계

| 한계 | 상세 내용 | 영향 |

### 7.3 AI(Text-to-SQL) 한계

> 출처: `[C-§3]` LLM 연동 설계

| 한계 | 상세 내용 | 대응 |
|------|----------|------|

| OD 쿼리 응답 시간 | 수분~수십분 | **5초 이내** | 즉시 응답 |
| Text-to-SQL 정확도 | N/A | 핵심 쿼리 **90%+** | 신규 |
| 보고서 작성 속도 | 기준 | **2배 향상** | 50% 단축 |

## 9. 용어 사전

| 용어 | 설명 |
|------|------|
| **KTDB** | 국가교통 데이터베이스. 국토교통부에서 운영하는 교통 데이터 플랫폼 |
| **OD** | Origin-Destination. 출발지(O)와 도착지(D) 간의 통행량 |
| **AAWDT** | Annual Average Weekday Daily Traffic. 연평균 **평일** 일통행량 (주말·공휴일 제외) |
| **AADT** | Annual Average Daily Traffic. 연평균 일통행량 (주말 포함) |
| **존(Zone)** | 교통 분석을 위해 행정구역을 번호로 관리하는 체계 |
| **파티셔닝** | 대용량 테이블을 물리적으로 여러 조각(파티션)으로 나누어 저장하는 기법 |
| **뷰(View)** | DB에서 자주 사용하는 조회 조건을 미리 정의해 둔 가상 테이블 |
| **Materialized View** | 뷰의 결과를 물리적으로 저장해 둔 것. 빠르지만 수동 갱신 필요 |
| **ETL** | Extract(추출), Transform(변환), Load(적재). 데이터를 소스에서 꺼내 가공하여 DB에 넣는 과정 |
| **PostgreSQL** | 오픈소스 관계형 데이터베이스 관리 시스템 |
| **Text-to-SQL** | 자연어 질문을 SQL 질의로 자동 변환하는 AI 기술 |
| **is_derived** | "파생 행" 표시 컬럼. TRUE이면 다른 행의 합산값 (합계 행), FALSE이면 원본 데이터 |
| **DDL** | Data Definition Language. 테이블 생성·변경 등 DB 구조를 정의하는 명령어 |
| **FK (외래 키)** | 다른 테이블의 값을 참조하는 컬럼. 잘못된 존 번호 입력 시 DB가 오류 발생시켜 데이터 무결성 보장 |
| **CP949** | 한글 Windows에서 사용하는 문자 인코딩. 수도권 TXT 파일이 이 인코딩으로 작성됨 |
| **LLM** | Large Language Model. 대규모 언어 모델 (ChatGPT, Claude 등) |
| **롱폼/와이드폼** | 데이터 저장 형태. 롱폼=한 행에 하나의 값, 와이드폼=한 행에 여러 수단/목적의 값 |
