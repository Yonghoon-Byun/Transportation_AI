## 개요

본 문서는 **KTDB(국가교통 데이터베이스)에서 제공하는 교통 원시 데이터를 PostgreSQL 데이터베이스에 적재하기 위한 스키마 설계**를 기술한다.

KTDB는 전국 여객·화물 통행량 OD 데이터, 사회경제지표, 접근수단 OD 등을 ZIP 형태로 배포하며, 본 프로젝트는 이를 자동으로 파싱·정제하여 DB에 적재하고, LLM 기반 자연어 질의 엔진을 통해 보고서용 성과품을 즉시 생성하는 플랫폼 구축을 목표로 한다.

본 설계서는 다음 독자를 대상으로 작성되었다.

- **교통 분야 실무자**: 어떤 원본 데이터가 어떤 테이블에 저장되는지, 어떤 컬럼으로 조회할 수 있는지 파악
- **개발 담당자**: 테이블 구조, 키 관계, 인덱스·파티셔닝 전략 등 구현에 필요한 사양 확인

> 본 문서는 설계 단계 문서로, 실제 구현 과정에서 세부 사항이 변경될 수 있다.
> SQL CREATE 코드 대신 **표 형식**으로 컬럼 구조를 기술하여 비개발직 열람 편의를 도모하였다.

## 1. 문서 범례

### 1-1. 표에서 사용하는 기호

| 기호 | 의미 |
|---|---|
| **PK** | 기본 키 (Primary Key) — 테이블에서 각 행을 유일하게 구별하는 컬럼. 중복·공백 불가 |
| **FK(테이블명)** | 외래 키 (Foreign Key) — 다른 테이블의 PK를 참조하는 컬럼. 참조 무결성 보장 |
| **UK** | 유일 키 (Unique Key) — 중복 불가, PK가 아닌 별도 유일성 제약 |
| **-** | 일반 컬럼 |

### 1-2. 데이터 타입 안내

| 표기 | 의미 |
|---|---|
| 소정수 | 정수, 최대 32,767까지 (SMALLINT) |
| 정수 | 정수, 최대 약 21억까지 (INTEGER) |
| 대정수 | 정수, 매우 큰 숫자 (BIGINT) |
| 소수(p,s) | 소수점 포함 실수. p=전체 자릿수, s=소수점 이하 자릿수 (NUMERIC) |
| 문자열(n) | 최대 n글자 문자열 (VARCHAR) |
| 고정문자(n) | 정확히 n글자 고정 문자열 (CHAR) |
| 긴문자열 | 길이 제한 없는 문자열 (TEXT) |
| 불리언 | 참(true) / 거짓(false) (BOOLEAN) |
| 일시 | 날짜+시각+시간대 (TIMESTAMPTZ) |
| 자동증가정수 | 새 행이 추가될 때 자동으로 1씩 증가하는 정수 (SERIAL) |

## 2. 핵심 개념 이해

### 존(Zone)이란?

KTDB 데이터는 특정 행정구역을 **번호로 관리**합니다. 이 번호 체계를 "존(Zone)"이라고 합니다.

- **전국 250존**: 전국 시군구를 250개 번호로 관리. 국가 단위 분석에 사용.
- **수도권 읍면동존 (1,310개)**: 수도권 읍면동을 1~1,310번으로 관리. 수도권 상세 분석에 사용.
- **비수도권 권역별 존**: 부산울산권·대구광역권 등 각 권역 내 읍면동을 별도 번호로 관리.

두 체계는 연계 키를 통해 서로 연결됩니다.

### OD 데이터란?

**O**rigin(출발지)와 **D**estination(도착지) 사이의 통행량을 나타냅니다. 250존 기준이면 250×250 = 62,500개의 출발-도착 조합이 존재합니다.

KTDB에서는 이 OD 데이터를 엑셀의 **행렬(매트릭스) 형태**로 제공하지만, DB에서는 검색 편의를 위해 **한 행이 하나의 OD쌍**이 되는 **세로형(Tall) 형태로 변환**하여 저장합니다.

```
[원본 엑셀 형태 — 매트릭스]        [DB 저장 형태 — 세로형]
     출발1  출발2  출발3  ...        출발존 | 도착존 | 연도 | 승용차 | 버스 ...
도착1  100    200    150  ...          1   |   1   | 2023 |  100   | 200
도착2  300     50    400  ...          1   |   2   | 2023 |  300   |  50
도착3  ...                             1   |   3   | 2023 |  ...
```

세로형으로 바꾸면 "2023년 A지역에서 B지역으로 가는 승용차 통행량은?" 같은 질의를 간단하게 처리할 수 있습니다.

## 3. 원본 데이터 → DB 테이블 매핑

### 3-1. 전체 흐름

```
[docs/data/ 폴더 — 21개 KTDB ZIP 파일]
              │
              │  ETL 자동 처리 (압축 해제 → 파싱 → 정제 → 적재)
              ▼
┌─────────────────────────────────────────────────────────────────────┐
│  ZIP 안의 파일 종류                      적재 DB 테이블             │
│                                                                     │
│  1. 존체계/250 존체계.xlsx           →  zone_250                   │
│  1. 존체계/존체계.xlsx (수도권)       →  zone_emd_metro             │
│  1. 존체계/존체계.xlsx (비수도권 5권역)→  zone_emd_region           │
│                                                                     │
│  3. 사회경제지표/*.xlsx (전국)        →  socio_national             │
│  3. 사회경제지표/*.xlsx (비수도권)    →  socio_region               │
│  3. 사회경제지표/인구수/*.TXT (수도권)→  socio_metro_pop            │
│  3. 사회경제지표/종사자수/*.TXT       →  socio_metro_work           │
│  3. 사회경제지표/취업자수/*.TXT       →  socio_metro_emp            │
│  3. 사회경제지표/학생수/*.TXT         →  socio_metro_student        │
│                                                                     │
│  5. OD/전국_주수단별OD(250).xlsx     →  od_national_mode           │
│  5. OD/전국_목적별OD(250).xlsx       →  od_national_purpose        │
│  4. OD/{권역명}_주수단통행.xlsx      →  od_region_mode             │
│  4. OD/{권역명}_목적통행.xlsx        →  od_region_purpose          │
│  4. OD/OD_MMODE_*.TXT (수도권)       →  od_metro_mode              │
│  4. OD/ODTRIP*_F.OUT (수도권)        →  od_metro_purpose           │
│  4. OD/OD_MMODE_{목적코드}_*.TXT    →  od_metro_purpose_mode      │
│                                                                     │
│  접근수단OD 배포용*.xlsx (전국)       →  od_access_national         │
│  {권역}_접근수단OD*.xlsx (4개 권역)  →  od_access_region           │
│                                                                     │
│  화물자동차OD_*.xlsx                 →  od_freight                 │
└─────────────────────────────────────────────────────────────────────┘
              │
              ▼
      [PostgreSQL DB]
```

### 3-2. ZIP 파일별 적재 테이블 대응표

| ZIP 데이터셋 | 파일 유형 | 적재 테이블 |
|---|---|---|
| FRE-CAR-00 화물자동차 OD | xlsx | `od_freight` |
| PSN-MOD-10 전국 250존 주수단 OD | xlsx | `zone_250`, `socio_national`, `od_national_mode` |
| PSN-MOD-11 수도권 주수단 OD | TXT(UTF-8) + TXT(CP949) | `zone_emd_metro`, `socio_metro_*`, `od_metro_mode` |
| PSN-MOD-12~16 권역별 주수단 OD | xlsx | `zone_emd_region`, `socio_region`, `od_region_mode` |
| PSN-MOD-20 전국 접근수단 OD | xlsx | `od_access_national` |
| PSN-MOD-21 수도권 목적×수단 OD | TXT(UTF-8) + TXT(CP949) | `socio_metro_*`, `od_metro_purpose_mode` |
| PSN-MOD-22~25 권역별 접근수단 OD | xlsx | `od_access_region` |
| PSN-OBJ-00 전국 250존 목적별 OD | xlsx | `zone_250`, `socio_national`, `od_national_purpose` |
| PSN-OBJ-01 수도권 목적별 OD | .OUT(UTF-8) + TXT(CP949) | `zone_emd_metro`, `socio_metro_*`, `od_metro_purpose` |
| PSN-OBJ-02~06 권역별 목적별 OD | xlsx | `zone_emd_region`, `socio_region`, `od_region_purpose` |

> 동일한 파일이 여러 ZIP에 중복 수록된 경우(존체계, 사회경제지표, 재차인원 등)는 ETL에서 중복 적재를 방지합니다.

## 4. 설계 원칙 요약

| 원칙 | 내용 |
|---|---|
| **연도는 행(row)으로** | 원본 엑셀의 연도별 컬럼(2023년, 2025년...)을 DB에서는 `year` 컬럼 한 개로 통합 |
| **수단/목적은 컬럼으로** | 승용차·버스·출근·귀가 등은 각각 별도 컬럼으로 저장 (조회 쿼리 단순화) |
| **NULL 대신 0 저장** | 해당 수단/목적이 없는 경우(예: 제주권 도시철도) 0.0으로 저장 |
| **권역별 OD는 단일 테이블로 통합** | `region_code` 컬럼으로 권역을 구분하여 테이블 수 최소화 |
| **수도권 대용량 테이블 파티셔닝** | 1,000만 행 이상인 수도권 OD 테이블은 연도별로 물리적 분리 |
| **소수점 3자리 보존** | 원본 데이터 소수점 그대로 보존 (`소수(12,3)` 타입 사용) |

## 6. 참조(코드) 테이블 상세

코드 테이블은 OD·사회경제지표 테이블에서 사용하는 분류 코드의 이름과 속성을 관리합니다. LLM이 "출근 통행량을 알려줘"라는 질의를 받았을 때 `purpose_code = 'commute'`를 찾아가는 방식으로 활용됩니다.

---

### 6-1. 권역 코드 (`ref_region`)

| 컬럼명 | 설명 | 타입 | 키 |
|---|---|---|---|
| region_code | 권역 번호 | 소정수 | PK |

> 원본 파일의 컬럼명은 `대존`(시도명), `소존`(시군구명)입니다.

## 17. 인덱스·파티셔닝 전략

### 파티셔닝 (대용량 테이블 분할)

파티셔닝이란 대용량 테이블을 물리적으로 여러 조각으로 나누어 저장하는 방식입니다. "2023년 데이터만 조회"할 때 다른 연도 파티션을 읽지 않아도 되므로 조회 성능이 크게 향상됩니다.

| 테이블 | 파티션 기준 | 파티션 수 | 이유 |
|---|---|---|---|
| `od_metro_mode` | 연도(year) | 7 | 12M행, 연도 단위 조회 빈번 |
| `od_metro_purpose` | 연도(year) | 7 | 12M행 |
| `od_metro_purpose_mode` | 연도(year) | 7 | **48M행**, 가장 큰 테이블 |
| `od_region_mode` | 권역(region_code) | 5 | 권역별 분석 쿼리 빈번 |
| `od_region_purpose` | 권역(region_code) | 5 | 동일 |
| `od_access_region` | 권역(region_code) | 4 | 권역별 접근수단 분석 |

> 전국 250존 OD는 437K행으로 파티셔닝 없이 인덱스만으로 충분합니다.

### 인덱스 원칙

1. **OD 테이블**: 출발존+연도, 도착존+연도 복합 인덱스 — "A 지역에서 출발하는 통행량" 조회 빠르게 지원
2. **사회경제지표**: 지표코드+연도+존번호 복합 인덱스 — "2030년 수도권 인구" 조회 지원
3. **존체계 테이블**: 행정기관코드 인덱스 — TXT 파일 ZONE_ID 매핑 시 빠른 검색

## 18. 설계 결정 사항 (Decision Log)

개발 과정에서 여러 방식 중 하나를 선택한 근거를 기록합니다.

| # | 결정 | 선택한 이유 | 검토한 대안 |
|---|---|---|---|
| D-01 | OD 테이블을 Wide(수단/목적 컬럼화)로 | LLM이 SQL 생성 시 쿼리가 단순해짐 | Tall: 수단을 행으로 → 행 수 7배 증가, GROUP BY 필요 |
| D-02 | 연도는 모두 행(row)으로 | 다연도 비교 쿼리 단순화 (`WHERE year IN (2023, 2030)`) | 연도를 컬럼으로: 연도 추가 시 스키마 변경 필요 |
| D-03 | 권역별 OD를 단일 테이블 통합 | 테이블 수 최소화, LLM 컨텍스트 단순화 | 권역별 테이블 분리: JOIN 불필요하나 테이블 10개 증가 |
| D-04 | 수도권 col08~col14 임시 보존 | 원본 무손실 보존 후 정의 확인 예정 | 미확인 컬럼 제외: 원본 복구 불가 |
| D-05 | 수도권 목적OD 합계 자동계산 컬럼 | 원본에 합계 없음, 수동 계산 시 오류 위험 | ETL에서 계산 후 저장: 가능하지만 자동계산이 더 안전 |
| D-06 | 수도권 인구 1세→5세 구간 집계 후 저장 | 204컬럼 → 45컬럼으로 축소, 쿼리 단순화 | 1세 단위 보존: 컬럼 수 과다, 상위 분석에 불필요 |
| D-07 | NULL 대신 0.0 저장 | 집계(SUM) 처리 시 NULL 예외처리 불필요 | NULL 사용: 미제공 구분 명확하나 쿼리 복잡 |
| D-08 | FK 참조 유지 | 잘못된 존 번호 적재 시 DB에서 즉시 오류 발생, 무결성 보장 | FK 제거: 적재 속도 향상되나 데이터 정합성 검증이 ETL에 의존 |

## 19. 부록: 주요 쿼리 패턴 예시

실무자가 자주 요청할 것으로 예상되는 조회 유형입니다. LLM은 이 패턴을 참고하여 SQL을 생성합니다.

### 예시 1. 특정 시군구의 목적별 통행량 (전국 OD 기준)

> "대전시 서구의 2023년 목적별 통행 발생량 표를 만들어줘"

```sql
SELECT
    '출근'    AS 목적, SUM(commute)     AS 통행량
FROM od_national_purpose
WHERE origin_250 = (SELECT zone_250_id FROM zone_250
                    WHERE sido_name = '대전광역시' AND sigungu_name = '서구')
  AND year = 2023
UNION ALL
SELECT '등교', SUM(school)     FROM od_national_purpose WHERE origin_250 = ... AND year = 2023
UNION ALL
SELECT '업무', SUM(business)   FROM od_national_purpose WHERE origin_250 = ... AND year = 2023
UNION ALL
SELECT '귀가', SUM(return_home) FROM od_national_purpose WHERE origin_250 = ... AND year = 2023
UNION ALL
SELECT '기타', SUM(other)      FROM od_national_purpose WHERE origin_250 = ... AND year = 2023;
```

### 예시 2. 권역별 수단별 통행량 변화 (2023 vs 2030)

> "부산울산권의 2023년 대비 2030년 수단별 통행량 변화 그래프 만들어줘"

```sql
SELECT year,
       SUM(car)   AS 승용차,
       SUM(bus)   AS 버스,
       SUM(rail)  AS 철도,
       SUM(metro) AS 도시철도
FROM od_region_mode
WHERE region_code = 2        -- 부산울산권
  AND year IN (2023, 2030)
GROUP BY year
ORDER BY year;
```

### 예시 3. 수도권 특정 행정동 발생 통행량 Top 10 도착지

> "강남구 역삼1동에서 출발하는 통행이 가장 많은 도착 행정동 10개 뽑아줘"

```sql
SELECT
    d.haengjeongdong_name AS 도착_행정동,
    SUM(om.car + om.bus + om.metro + om.taxi) AS 총통행량
FROM od_metro_mode om
JOIN zone_emd_metro o ON o.seq_no = om.o_seq
JOIN zone_emd_metro d ON d.seq_no = om.d_seq
WHERE om.year = 2023
  AND o.sigungu_name = '강남구'
  AND o.haengjeongdong_name = '역삼1동'
GROUP BY d.haengjeongdong_name
ORDER BY 총통행량 DESC
LIMIT 10;
```
