## 1. 왜 데이터베이스가 필요한가

현재 KTDB 데이터 활용 업무는 **ZIP 다운로드 → Excel 수작업 분석 → 보고서 작성** 과정을 거칩니다.

| 문제 | 구체적 상황 |
|------|------------|
| 파일이 너무 많다 | ZIP 25건, 내부 파일 수백 개 |
| 포맷이 제각각 | Excel(.xlsx)과 텍스트(.OUT, .TXT) 혼재, 권역마다 다름 |
| 데이터 양이 방대 | 전체 약 **9,300만 행** (Excel 한계 100만 행 초과) |
| 수단 명칭이 권역마다 다름 | 같은 "지하철"이 `지하철`, `도시철도` 등으로 표기 |
| 합계 행 이중 계산 위험 | "합계" 행이 포함되어 SUM 시 2배 오류 발생 가능 |

DB를 구축하면 **자동 파싱·정제 → 통합 저장 → AI 자연어 질의 → 5초 내 결과**가 가능합니다.

## 2. KTDB 원본 데이터 개요

### 2.1 데이터 분류

| 대분류 | 소분류 | 설명 | ZIP 수 |
|--------|--------|------|--------|
| 여객(PSN) | 목적별(OBJ) | 출근·등교·업무·귀가·기타별 통행량 | 7건 |
| 여객(PSN) | 수단별(MOD-1x) | 승용차·버스·철도 등 수단별 통행량 | 7건 |
| 여객(PSN) | 접근수단 등(MOD-2x) | 철도역 접근수단, 목적×수단 OD | 7건 |
| 화물(FRE) | 자동차(CAR) | 소·중·대형 화물차 톤급별 통행량 | 1건 |

### 2.2 권역별 파일 형태 차이 (핵심 난제)

같은 종류의 데이터인데도 **권역에 따라 파일 형태가 완전히 다릅니다.**

| 구분 | 전국 250존 | 비수도권 5개 권역 | 수도권 |
|------|-----------|-------------------|--------|
| 파일 포맷 | Excel (.xlsx) | Excel (.xlsx) | **텍스트 (.OUT, .TXT)** |
| 인코딩 | UTF-8 | UTF-8 | **CP949 (한글 전용)** |
| 헤더(열 이름) | 있음 | 있음 | **없음** |
| 존코드 형태 | 정수 1~250 | 정수 1~N | **7자리 문자열** |
| 파생 컬럼 | 없음 | 있음 (승용차/택시 합산 등) | 없음 |

### 2.3 데이터 규모

| 데이터 종류 | 예상 행 수 |
|-------------|-----------|
| 전국 250존 여객 OD | 약 87만 행 |
| 비수도권 권역별 OD | 약 1,850만 행 |
| **수도권 OD** | **약 7,200만 행 (전체의 77%)** |
| 화물 OD + 사회경제지표 + 기타 | 약 200만 행 |
| **합계** | **약 9,300만 행** |

## 3. 핵심 개념 이해

### 3.1 존(Zone) 체계

KTDB는 행정구역을 번호 체계로 관리합니다.

```
17존 (시도)        예: 1=서울, 2=부산, ... 17=제주
  └── 250존 (시군구)   예: 1=종로구, 2=중구, ...
       └── 소존 (읍면동)   수도권 1,310개, 비수도권 권역별 수백 개
```

- 비수도권 소존 번호는 **권역마다 1번부터 시작** → 권역코드 + 소존번호를 함께 써야 구분 가능

### 3.2 OD 데이터

OD(Origin-Destination) = 출발지→도착지 통행량. 원본은 행렬(매트릭스) 형태이지만, DB에서는 검색에 유리한 **세로형(롱폼)** 으로 변환하여 저장합니다.

```
[원본 엑셀: 행렬]                [DB: 세로형]
      도착1  도착2 ...            출발존 | 도착존 | 연도 | 승용차 | 버스
출발1  100    200 ...               1   |   1   | 2023 |  100  | 200
출발2  300     50 ...               1   |   2   | 2023 |  300  |  50
```

### 3.3 수단 분류 체계의 차이

**같은 교통수단이 권역마다 다른 이름**으로 표기되는 것이 가장 까다로운 문제입니다.

| 교통수단 | 전국 250존 | 비수도권 | 수도권 |
|----------|-----------|---------|--------|
| 도시철도 | `지하철` | `도시철도` | 열 이름 없음 |
| 일반+고속철도 | 분리 제공 | `일반/고속철도` (합산) | 열 이름 없음 |
| 도보+자전거 | 없음 | `도보/자전거` | 열 이름 없음 |
| 택시 | 없음 | `택시` | 열 이름 없음 |
| 항공·해운 | 분리 제공 | 없음 (기타에 포함 추정) | 불명 |

**해결**: DB에 "동의어 매핑 테이블"을 만들어 "지하철=도시철도" 자동 변환.

### 3.4 파생 행(합계) 이중 계산 문제

원본에 "합계" 행이 원본 행과 함께 있어 SUM 시 **2배 오류** 발생.

```
출근:  100  ← 원본        해결: 모든 행에 is_derived(파생여부) 표시.
등교:   50  ← 원본              AI 분석 뷰에서 원본만 자동 필터.
합계:  150  ← 파생값
SUM = 300  ← ❌ (정답은 150)
```

## 4. 구축된 DB의 전체 구조

### 4.1 데이터 처리 흐름

```
[KTDB ZIP 25건] → [자동 파싱·정제] → [PostgreSQL DB 약 9,300만 행] → [AI 자연어 질의] → [즉시 결과]
```

### 4.2 테이블 구성

| 종류 | 테이블 수 | 역할 | 행 수 |
|------|----------|------|-------|
| 참조 테이블 | 6개 | 코드의 이름·속성 관리 | 수십 행 |
| 존체계 테이블 | 3개 | 존 번호 ↔ 행정구역명 매핑 | ~3,800행 |
| **OD 테이블** | **10개** | **핵심 통행량 데이터** | **~9,200만 행** |
| 사회경제지표 | 6개 | 인구·종사자수·학생수 등 | ~16만 행 |
| 운영 테이블 | 2개 | 데이터 적재 이력 관리 | 수백 행 |

### 4.3 테이블 간 관계

```
[참조 테이블: 권역코드, 연도, 목적코드, 수단코드]
         │ (코드 참조)
[존체계 테이블]  ← "종로구는 250존 1번" 같은 매핑 정보
         │ (존번호로 연결)
[OD 테이블]  ← "1번 존 → 34번 존, 출근, 100통행/일"
         │ (같은 존번호로 연결)
[사회경제지표]  ← "1번 존, 2023년, 총인구 = 123,456명"
```

### 4.4 대용량 처리: 파티셔닝

대용량 테이블을 연도 또는 권역 기준으로 **물리적으로 분할**하여 조회 성능을 확보합니다.

| 테이블 | 파티션 기준 | 파티션 수 | 효과 |
|--------|-----------|----------|------|
| 수도권 OD 3종 | 연도 | 각 7개 | 1,200만~4,800만 행 → 연도별 170만~690만 행만 스캔 |
| 비수도권 OD 2종 | 권역 | 각 5개 | 권역별 분리 조회 |
| 전국 250존 OD 3종 | 연도(RANGE) | 각 7개 | "2023~2030년" 범위 조회 최적화 |

## 5. AI(자연어 질의) 연동

### 5.1 작동 원리

```
사용자: "2023년 서울에서 부산으로 출근 통행량은?"
    ↓
AI가 SQL 자동 생성 → DB 실행 → 결과 반환
    ↓
"서울→부산 출근 통행량은 일평균 12,345통행(AAWDT 기준)입니다."
```

### 5.2 AI 전용 분석 뷰

AI는 원본 테이블을 직접 보지 않고, **"분석 뷰"** 를 통해서만 데이터에 접근합니다.

| 뷰 | 기능 |
|----|------|
| 목적OD/수단OD 뷰 | 합계 행 자동 제외 + 지역 이름 자동 연결 |
| 17존 집계 뷰 | 시도 수준 사전 집계 (즉시 응답) |
| 지명 검색 뷰 | "해운대구" → 존 번호 자동 검색 |
| 수단 매핑 테이블 | "지하철=도시철도" 자동 변환 |

### 5.3 성능 목표

| 단계 | 목표 시간 |
|------|----------|
| AI SQL 생성 | 2~3초 |
| DB 쿼리 실행 | 1~2초 |
| 결과 포맷팅 | 0.3초 |
| **전체** | **5초 이내** |

## 6. 구축에 필요한 지식과 역할 분담

### 6.1 필요 지식

| 영역 | 난이도 | 설계엔지니어가 알아야 할 수준 |
|------|--------|--------------------------|
| KTDB 데이터 구조 (존체계, OD) | ★★☆ | 어떤 데이터가 어디에 있는지 파악 |
| 수단 분류 체계 차이 | ★★★ | 같은 수단의 다른 이름 인지 |
| 데이터 단위 (AAWDT, 통행/일 vs 대/일) | ★★☆ | 보고서 단위 정확 기재 |
| SQL 기본 | ★★☆ | AI가 대신 생성, 결과 해석만 |
| PostgreSQL 운영 | ★★★★ | **개발자 영역** |
| ETL 파이프라인 | ★★★★★ | **개발자 영역** |

### 6.2 역할 분담

| 역할 | 담당자 | 주요 업무 |
|------|--------|----------|
| 데이터 검증자 | 설계엔지니어 | 적재 데이터 검증, 분석 결과 해석 |
| DB 구축자 | 개발자/DBA | PostgreSQL, DDL, ETL 파이프라인 |
| AI 연동 개발자 | 개발자 | Text-to-SQL, 뷰 레이어, 보고서 자동화 |
| 도메인 전문가 | 교통공학 전공자 | 수단 분류 검증, 분석 시나리오 정의 |

### 6.3 설계 결정이 필요한 사항

DB 구축 전에 **설계엔지니어의 판단이 필요한 항목**들입니다.

| 결정 사항 | 선택지 | 판단 이유 |
|----------|--------|----------|
| 수단 분류 통합 여부 | 통합 정규화 / 원본 유지 | 전국 vs 권역 비교 가능성 |
| 파생 컬럼(승용차/택시 합산) 포함 | 제외(중복) / 포함(원본 보존) | 합산값 직접 사용 여부 |
| 17존 피벗 시트 활용 | 요약 테이블 저장 / 무시 | 시도간 요약표 필요성 |
| 접근수단 OD 포함 범위 | 포함 / 1차 제외 | 접근수단 분석 우선순위 |
| Excel 수식 셀 처리 | 캐시값 읽기 / 직접 계산 | 수식 오류 가능성 |
| 과거년도 데이터 포함 | 포함(버전 구분) / 최신만 | 과거 버전 비교 필요성 |

## 7. 한계점과 주의사항

### 7.1 데이터 한계

| 한계 | 영향 | 대응 |
|------|------|------|
| 수도권 TXT 열 이름 부분 미확인 | 10개 수단 컬럼 중 7개는 비수도권 대응으로 추정 완료(도보/자전거, 승용차, 버스, 일반철도, 고속철도, 도시철도, 택시), **col12~14 3개는 여전히 불명** | 설명자료(.hwp)로 추정값 검증 및 불명 3개 확인 필요 |
| 수도권 "업무" 목적 누락 | 업무 목적의 수단별 분석 불가 | KTDB에 의도적 누락 여부 확인 필요 |
| 수단 분류 3종 혼재 | 전국 단위 완벽한 통합 비교 불가 | 동의어 매핑으로 부분 해결 |
| 사회경제지표 권역별 차이 | 인구 시트명이 `5-24세` vs `5-29세`로 다름 | 지표코드 매핑 규칙 정의 완료 |
| 화물 OD 컬럼명 오류 | 도착 17존 컬럼명이 출발과 동일 | 위치 기반 파싱으로 처리 |

> **보고서 작성 시 단위 주의**:
> - **재차인원**: 통행/일 → 대/일 환산 (예: 승용차 재차인원 1.3명 → 100통행은 약 77대)
> - **AADT 전환계수**: 주중 평균(AAWDT) → 연평균(AADT) 환산

### 7.2 기술적 한계

| 한계 | 영향 |
|------|------|
| Azure PostgreSQL 스키마 제한 | `public` 스키마에 `ktdb_` 접두사 사용 |
| 9,300만 행 초기 적재 | 수시간 소요 예상 |
| Materialized View 갱신 | 데이터 변경 시 수동 REFRESH 필요 |

### 7.3 AI 한계

| 한계 | 대응 |
|------|------|
| LLM 모델 미확정 | 모델-agnostic 설계로 유연성 확보 |
| 복잡한 다중 조건 질의 정확도 한계 | 정형 쿼리 템플릿 사전 정의 |
| "수단별 OD"와 "접근수단 OD" 혼동 위험 | 접근수단 OD는 AI에 미노출 |
